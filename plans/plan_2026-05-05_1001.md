## Analysis

The 500 error on `POST /api/chat/send` is caused by **cascading infrastructure failures** in the backend startup chain. The backend's `start()` function requires all three services to initialize successfully:

1. **`initDatabase()`** — Connects to PostgreSQL at `localhost:5432`
2. **`initRedis()`** — Connects to Redis at `localhost:6379`
3. **`initQueue()`** — Creates a BullMQ queue (also requires Redis)

If **any** of these fail (e.g., PostgreSQL or Redis aren't running), the server either crashes on startup (`process.exit(1)`) or if it somehow starts partially, the `processChat()` function in `modelRouter.ts` calls `getCached()` which uses the Redis client — resulting in a 500 error.

Even if the server starts, the chat route hits the database via `findOrCreateUser()`, `getChatSession()`, and `createChatSession()` — all of which will throw if PostgreSQL isn't accessible.

Additionally, the Gemini API call uses model `gemini-2.5-pro-preview-05-06` which may not be available or may require a different API key configuration.

## Root Causes

1. **PostgreSQL not running** — The backend connects to `postgresql://vibe_user:vibe_password@localhost:5432/vibe_coding`. If you're not running Docker Compose, this DB doesn't exist.
2. **Redis not running** — The cache and queue services require Redis at `localhost:6379`.
3. **No graceful degradation** — If DB/Redis aren't available, the entire request fails with a 500.

## Fix Plan

### Step 1: Make Redis/Cache optional (graceful degradation)
Modify `cache.ts` to handle Redis unavailability — `getCached()` returns null, `setCache()` is a no-op. This lets chat work without Redis.

### Step 2: Make the Queue optional
Modify `queue.ts` so `initQueue()` doesn't crash the server if Redis is down.

### Step 3: Make the Database optional with an in-memory fallback
Add a simple in-memory store in `db/index.ts` so that when PostgreSQL is unavailable, the app still works for local development (messages are stored in memory).

### Step 4: Add error handling to `modelRouter.ts`
Wrap the Gemini/GPT/Hermes calls with better error messages so you can see exactly what's failing.

### Step 5: Fix the frontend `page.tsx` error handling
Currently on line 121, if the AI call fails, the user sees "Something went wrong" — add the actual error details for debugging.

### Step 6: Verify API keys are valid
The Gemini model name `gemini-2.5-pro-preview-05-06` might not match your SDK version. We should add a fallback model name.
